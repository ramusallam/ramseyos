# RamseyOS Shell Navigation

## Overview

The shell is the persistent frame around all content in RamseyOS. It provides navigation, context, and quick actions that remain available regardless of which view the user is in. The shell is workspace-agnostic — it aggregates across all workspaces and provides entry points into workspace-specific views.

## Primary Shell Areas

The shell has six primary areas, listed in recommended navigation order:

### 1. Today Dashboard (`/dashboard`)

The home screen. Shows the Daily Card, approvals, calendar glance, quick capture, and reflection layer. This is where the user starts every session.

**Why it's first:** It answers the most urgent question — "What am I doing right now?" Everything else is secondary to the current day's plan.

### 2. Universal Inbox (`/inbox`)

A capture and processing queue for incoming items: tasks, ideas, notes, requests. Items land here when captured via Quick Capture or external integrations. The user processes the inbox by assigning items to workspaces, converting them to tasks, or archiving them.

**Why it's second:** After reviewing the day's plan, the next action is often processing what came in since the last session.

### 3. Tasks / Projects (`/tasks`)

Full task management: create, edit, filter, and organize tasks and projects across all workspaces. Supports views by workspace, by project, by status, and by due date. This is the operational backbone — where tasks are managed outside the daily card context.

**Why it's third:** Task management is a planning activity, not a daily execution activity. It supports the dashboard but doesn't replace it.

### 4. Approval Queue (`/approvals`)

All AI-generated drafts awaiting review. Each approval shows the source workflow, workspace, content preview, and actions (approve, reject, revise). The queue enforces the review-before-publish principle.

**Why it's fourth:** Approvals are generated by workflows and accumulate over time. They need regular attention but aren't the first thing to check each day.

### 5. Calendar (`/calendar`)

Full calendar view: daily, weekly, monthly. Shows all calendar items across workspaces. Supports future Google Calendar integration. Used for planning and scheduling, not for daily execution (that's the Daily Card's job).

**Why it's fifth:** The calendar is a reference tool. The Daily Card already surfaces today's events, so the full calendar is for planning ahead.

### 6. Admin / Settings (`/settings`)

Configuration panels for the system: workspace settings, workflow registry browser, model assignments, prompt template management, and system preferences (dark mode, notification timing, etc.).

**Why it's last:** Settings are rarely accessed. They should be reachable but not prominent.

## Navigation Structure

```text
Shell Side Nav
├── Today Dashboard     /dashboard
├── Universal Inbox     /inbox
├── Tasks / Projects    /tasks
├── Approval Queue      /approvals
├── Calendar            /calendar
├── ─────────────────
├── Sonoma Academy      /workspaces/sonoma
├── Concordia           /workspaces/concordia
├── Woven               /workspaces/woven
├── Cycles of Learning  /workspaces/cycles
├── ─────────────────
└── Admin / Settings    /settings
```

The nav is divided into three groups by a visual separator:

1. **Shell sections** (top) — Cross-workspace tools for daily operation.
2. **Workspaces** (middle) — Entry points into workspace-specific views.
3. **Admin** (bottom) — System configuration.

## Shell Versus Workspace

### What belongs in the shell

Anything that aggregates across workspaces or serves the user's daily rhythm:

- Today Dashboard (all workspaces' tasks and events)
- Universal Inbox (items from any source)
- Tasks / Projects (filterable by workspace, but shows all by default)
- Approval Queue (approvals from any workflow)
- Calendar (events from all contexts)
- Quick Capture (workspace-agnostic input)
- Reflection (personal, not workspace-specific)

### What belongs in workspaces

Anything specific to a single professional domain:

- Sonoma Academy: teaching tools, resource library, instructional support toolkit, admin workflows
- Concordia: discussion board tools, course management, rubric grading
- Woven: workshop builder, strategy documents, invoicing
- Cycles of Learning: blog editor, keynote builder, outreach tools

Workspace views are accessed by clicking the workspace name in the nav. Each workspace has its own internal navigation for its sub-areas.

When the user enters a workspace, the shell remains visible but the main content area shows workspace-specific navigation:

Workspace sub-navigation appears as a secondary nav within the main content area (e.g., tabs or a secondary sidebar), not in the primary shell nav. The shell nav always shows the top-level workspace entries only.

## Top Bar

The top bar is a thin persistent bar above the main content area:

- **Left:** Breadcrumb showing current location (e.g., "Today Dashboard" or "Sonoma > Teaching > Lesson Planning")
- **Center:** Quick Capture input (always accessible)
- **Right:** Approval count badge, user avatar/settings shortcut

The top bar does not change between shell sections and workspace views. It provides continuity.

## Responsive Behavior

- **Desktop (1024px+):** Side nav is visible and collapsed (icons only). Expands on hover or pin. Full top bar.
- **Tablet (768-1023px):** Side nav hidden, accessible via hamburger. Top bar compressed.
- **Mobile (< 768px):** Bottom tab bar with 4-5 key sections (Dashboard, Inbox, Tasks, Approvals, More). Top bar becomes a minimal header. Workspace access via "More" tab.

Mobile layout is designed but not implemented in Phase 1. The navigation structure supports it without data model changes.

## Keyboard Navigation (Future)

- `Cmd+K` or `/` — Command palette for quick navigation and actions
- `1-6` — Jump to shell sections by number
- `Cmd+N` — New task via quick capture
- Arrow keys — Navigate lists and cards

Keyboard shortcuts are planned but not implemented in Phase 1.
